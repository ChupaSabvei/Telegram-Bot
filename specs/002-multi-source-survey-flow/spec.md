# Feature Specification: Мульти-источники и опросный подбор

**Feature Branch**: `002-multi-source-survey-flow`

**Created**: 2026-06-14

**Status**: Draft

**Input**: User description: "Расширить источники данных (Timepad, МТПП, T-Bank Афиша, MTS Live, mos.ru, Time Out, Мой спортивный район и др.) и изменить UX: после /start — Случайный вариант / Опрос / Популярное; опрос по шагам (компания, категория активности, бюджет, формат/погода) с персонализированной карточкой результата и кнопками «Другой вариант», «В избранное», «Заново»."

## Clarifications

### Session 2026-06-14

- Q: Как совмещать новый опросный UX со старым функционалом 001 (категории, ИИ-поиск)? → A: При запуске — выбор города; затем пошаговый опрос как в исходном ТЗ; после всех шагов — «Другой вариант», «В избранное», «Заново»; произвольное текстовое сообщение в чат (не быстрый ответ на текущий шаг опроса) активирует ИИ-помощника по запросу пользователя.
- Q: Что показывать сразу после выбора города? → A: Главное меню из четырёх кнопок (🎲 Случайный \| 📝 Опрос \| 🔥 Популярное \| ❤️ Избранное); опрос начинается только по нажатию «📝 Опрос».
- Q: Как открыть сохранённое избранное? → A: Четвёртая кнопка «❤️ Избранное» на главном меню (в составе четырёхкнопочного меню).
- Q: Что делать, если под фильтры опроса ничего не найдено? → A: Сообщение «ничего не найдено» + кнопки «Смягчить бюджет», «Сменить формат», «🔄 Заново».
- Q: Как показывать «🔥 Популярное»? → A: Список из 5 пунктов + кнопка «Показать ещё» (до 10 всего).

### Session 2026-06-14 (follow-up)

- Q: Как опрос работает с событиями без категории активности (`activity_slug`) из источника? → A: Если в данных источника нет категории активности, при ingest подключается ИИ-помощник: ему передаётся описание события (название, описание, источник), и он определяет `activity_slug` для опросного подбора.
- Q: ИИ недоступен при синхронизации — что делать с событием без категории? → A: Rule-based fallback (ключевые слова, источник); если не вышло — сохранить как «не классифицировано», повторить ИИ/rule-based при следующей sync; sync не останавливается.
- Q: Кнопка «⚙️ Настройки» на главном меню? → A: Да — пятая кнопка «⚙️ Настройки» под четырьмя основными (смена города).

### Session 2026-06-12 (artifact sync)

- Q: Синхронизировать plan/tasks/contracts с FR-021/023 и ⚙️ Настройки после analyze? → A: Да — plan, tasks (T012–T014, T020), data-model, research, contracts обновлены; spec без новых требований.

### User Story 1 — Опросный подбор (Priority: P1)

Пользователь после `/start` сначала выбирает город (если ещё не выбран), затем «📝 Опрос» и проходит короткий сценарий из четырёх шагов: с кем идёт, категория активности, бюджет, формат (крытое / любое / у дома). На каждом шаге ответ даётся только кнопками быстрого выбора. Бот показывает одну персонализированную карточку с учётом всех ответов.

**Why this priority**: Это новый основной сценарий discovery, который заменяет «угадывание категории» и даёт релевантный результат для семьи, пары, компании друзей и т.д.

**Independent Test**: Пройти опрос «Семья → Семейный отдых → до 3000₽ → Всё равно» и получить карточку с заголовком контекста, описанием, ценой, форматом, адресом и тремя кнопками действий.

**Acceptance Scenarios**:

1. **Given** пользователь выбрал город, **When** бот показывает главный экран, **Then** доступны четыре основные кнопки: «🎲 Случайный вариант», «📝 Опрос», «🔥 Популярное», «❤️ Избранное», и кнопка «⚙️ Настройки» для смены города (опрос не начинается автоматически).
2. **Given** пользователь на главном экране после `/start`, **When** нажимает «📝 Опрос», **Then** бот показывает шаг «Кто идёт?» с вариантами: Один, Пара, Семья, Друзья.
3. **Given** пользователь выбрал «👨‍👩‍👧‍👦 Семья», **When** подтверждает выбор, **Then** бот показывает шаг категории активности: Спорт, Для детей, Семейный отдых, Культура, Гастро, Релакс.
4. **Given** пользователь выбрал «🏕 Семейный отдых», **When** подтверждает, **Then** бот показывает шаг бюджета: Бесплатно, до 1000₽, до 3000₽, Без лимита.
5. **Given** пользователь выбрал «💰 до 3000₽», **When** подтверждает, **Then** бот показывает шаг формата: Только крытое, Всё равно, У дома.
6. **Given** пользователь завершил все шаги опроса, **When** бот формирует результат, **Then** карточка содержит: персональный заголовок (аудитория + бюджет), название, 2–4 пункта описания, цену/«бесплатно», указание крытое/улица, адрес со ссылкой на навигатор, кнопки «➡️ Другой вариант», «❤️ В избранное», «🔄 Заново».
7. **Given** пользователь нажимает «➡️ Другой вариант», **When** в базе есть ещё подходящие варианты, **Then** бот показывает следующую карточку с теми же фильтрами опроса, не повторяя уже показанные.
8. **Given** пользователь нажимает «🔄 Заново», **When** действие обработано, **Then** опрос начинается с шага «Кто идёт?».

### User Story 1b — ИИ-помощник на свободный текст (Priority: P1)

Пользователь в любой момент (вне активного шага опроса с кнопками быстрого выбора) отправляет произвольное текстовое сообщение — бот активирует ИИ-помощника и отвечает по смыслу запроса (мероприятия, места, рекомендации).

**Why this priority**: Сохраняет гибкий поиск из фичи 001 без отдельной команды; пользователь может спросить «куда сходить с детьми» или «бильярд рядом» в свободной форме.

**Independent Test**: На главном экране (не на шаге опроса) отправить «концерт на выходных» — получить релевантный ответ ИИ-помощника в течение 10 секунд.

**Acceptance Scenarios**:

1. **Given** пользователь не на шаге опроса с кнопками быстрого выбора, **When** отправляет текстовое сообщение, **Then** ИИ-помощник обрабатывает запрос и возвращает ответ (мероприятия, места или пояснение, если ничего не найдено).
2. **Given** пользователь на шаге опроса «Кто идёт?» с кнопками, **When** нажимает кнопку «Семья», **Then** опрос продолжается; ИИ-помощник не вызывается.
3. **Given** пользователь на шаге опроса с кнопками, **When** вместо кнопки пишет произвольный текст, **Then** ИИ-помощник обрабатывает текст как запрос (текущий черновик опроса не сохраняется).

### User Story 2 — Агрегация из множества источников (Priority: P1)

Система наполняет базу мероприятий и мест из расширенного набора публичных афиш и городских сервисов, чтобы опрос, «Популярное» и «Случайный вариант» имели достаточный выбор.

**Why this priority**: Без новых источников опрос и новые категории (спорт, семейный отдых, гастро) будут возвращать пустые или нерелевантные результаты.

**Independent Test**: После синхронизации для Москвы в базе есть события минимум из 5 различных источников; каждое событие имеет название, дату (или период), место, ссылку на источник и признак платности где доступно.

**Acceptance Scenarios**:

1. **Given** запланирована синхронизация, **When** она завершается для Москвы, **Then** в базе появляются новые события из источников: Timepad, календарь МТПП, T-Bank Город, MTS Live, портал «Культура Москвы», Time Out, «Мой спортивный район», а также ранее подключённые агрегаторы; у событий без категории в источнике `activity_slug` определён ИИ-классификацией (FR-021).
2. **Given** одно и то же мероприятие присутствует в двух источниках, **When** выполняется синхронизация, **Then** пользователю показывается одна карточка без дубликатов.
3. **Given** источник временно недоступен, **When** синхронизация выполняется, **Then** остальные источники обновляются; сбой фиксируется, бот продолжает работать на доступных данных.
4. **Given** событие из источника без явной категории активности, **When** выполняется синхронизация и ИИ доступен, **Then** событию присваивается одна из 6 категорий опроса и сохраняется в базе.

### User Story 3 — Случайный вариант (Priority: P2)

Пользователь с главного экрана выбирает «🎲 Случайный вариант» и сразу получает одну карточку подходящего мероприятия или места в своём городе без прохождения опроса.

**Why this priority**: Быстрый вход для пользователей без конкретного запроса; повышает вовлечённость.

**Independent Test**: Нажать «🎲 Случайный вариант» три раза подряд — каждый раз новая карточка (пока есть данные), формат карточки как в US1.

**Acceptance Scenarios**:

1. **Given** пользователь на главном экране, **When** нажимает «🎲 Случайный вариант», **Then** в течение 5 секунд получает одну карточку с кнопками «Другой вариант», «В избранное», «Заново» (где «Заново» возвращает на главный экран).
2. **Given** в городе нет доступных событий, **When** пользователь запрашивает случайный вариант, **Then** бот сообщает об этом и предлагает пройти опрос или сменить город.

---

### User Story 4 — Популярное (Priority: P2)

Пользователь выбирает «🔥 Популярное» и получает подборку из наиболее востребованных мероприятий в городе (по данным источников: близкая дата, высокая заметность на афишах, повторяемость в нескольких источниках).

**Why this priority**: Даёт быстрый ориентир «куда идут другие» без настройки фильтров.

**Independent Test**: Нажать «🔥 Популярное» — получить список из 5 популярных вариантов с кнопкой «Показать ещё»; открыть любой пункт — полная карточка.

**Acceptance Scenarios**:

1. **Given** пользователь на главном экране, **When** нажимает «🔥 Популярное», **Then** бот показывает нумерованный список из 5 популярных вариантов на ближайшие 30 дней и кнопку «Показать ещё» (если доступно больше 5).
2. **Given** пользователь видит первые 5 популярных вариантов, **When** нажимает «Показать ещё», **Then** бот дополняет список до 10 пунктов (или до исчерпания доступных).
3. **Given** пользователь выбрал пункт из списка популярного, **When** открывает его, **Then** видит полную карточку в едином формате с кнопками «➡️ Другой вариант», «❤️ В избранное», «🔄 Заново».

---

### User Story 5 — Избранное (Priority: P3)

Пользователь сохраняет понравившиеся карточки через «❤️ В избранное» и может вернуться к ним позже.

**Why this priority**: Улучшает удержание, но не блокирует запуск опроса и новых источников.

**Independent Test**: Добавить карточку в избранное, перезапустить бота, открыть избранное — карточка на месте.

**Acceptance Scenarios**:

1. **Given** пользователь на карточке результата, **When** нажимает «❤️ В избранное», **Then** бот подтверждает сохранение.
2. **Given** у пользователя есть избранное, **When** он нажимает «❤️ Избранное» на главном меню, **Then** видит сохранённые карточки с актуальностью (прошедшие помечаются как «завершилось»).

---

### Edge Cases

- Пользователь прерывает опрос на середине — при следующем «Опрос» начинается сначала; черновик не сохраняется.
- Нет результатов под выбранные фильтры — бот показывает сообщение «ничего не найдено» и кнопки: «Смягчить бюджет», «Сменить формат», «🔄 Заново»; автоматическое смягчение фильтров без согласия пользователя не выполняется.
- Выбран «🏠 У дома» — показываются онлайн-трансляции, домашние активности и события с дистанционным форматом, если они есть в источниках; иначе честное сообщение и предложение сменить формат.
- Выбран «🌧 Только крытое» — исключаются чисто outdoor-события без крытой площадки.
- Событие без цены в источнике — в карточке «уточняйте на сайте».
- Событие без точного адреса — район или название площадки + ссылка на источник.
- Событие без `activity_slug` после ingest — сначала rule-based, затем ИИ при sync; при неудаче — «не классифицировано», повтор при следующей sync; до классификации не участвует в опросе (MAY — в «Случайном» и «Популярном»).
- ИИ недоступен при sync — rule-based only; sync продолжается; неклассифицированные события повторно обрабатываются позже (FR-023).
- Источник отдаёт только деловые/B2B-мероприятия (например, МТПП) — ИИ или rule-based classifier назначает «культура»; они не доминируют в «Семейном отдыхе», если не релевантны.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: После `/start` бот MUST сначала определить город пользователя (выбор из списка или сохранённый город из профиля); только после выбора города показывается главное меню с четырьмя основными опциями («🎲 Случайный вариант», «📝 Опрос», «🔥 Популярное», «❤️ Избранное») и кнопкой «⚙️ Настройки» для смены города.
- **FR-002**: При каждом `/start`, если город не сохранён, бот MUST запросить город до показа главного меню или опроса.
- **FR-003**: Опрос MUST включать 4 последовательных шага: аудитория (один / пара / семья / друзья), категория активности (спорт / для детей / семейный отдых / культура / гастро / релакс), бюджет (бесплатно / до 1000₽ / до 3000₽ / без лимита), формат (только крытое / всё равно / у дома).
- **FR-004**: Карточка результата MUST отображать: персональный заголовок с контекстом опроса, название, краткое описание списком, цену, indoor/outdoor-признак, адрес или локацию со ссылкой на навигатор, ссылку на первоисточник.
- **FR-005**: На карточке MUST быть кнопки: «➡️ Другой вариант», «❤️ В избранное», «🔄 Заново».
- **FR-006**: «Другой вариант» MUST подбирать следующее событие/место с теми же фильтрами опроса, не повторяя ранее показанные в текущей сессии.
- **FR-007**: Система MUST агрегировать события из минимум следующих публичных источников для Москвы: Timepad (включая afisha.timepad.ru), календарь МТПП (mostpp.ru), T-Bank Город Афиша, MTS Live, портал «Культура Москвы» (mos.ru), Time Out Москва, «Мой спортивный район» (moysportrayon.sport.mos.ru), KudaGo, Яндекс Афиша.
- **FR-008**: Для городов MVP кроме Москвы система MUST продолжать работать на уже подключённых агрегаторах; новые источники MAY подключаться поэтапно с приоритетом Москвы.
- **FR-009**: Синхронизация MUST выполняться по расписанию и вручную; каждый источник MUST иметь статус последней успешной/неуспешной загрузки.
- **FR-010**: Дедупликация MUST объединять одно и то же мероприятие из разных источников в одну пользовательскую карточку.
- **FR-011**: Подбор по опросу MUST учитывать: аудиторию (например, «для детей» / «семья»), категорию активности, бюджет, indoor/outdoor и горизонт 30 дней.
- **FR-012**: «Случайный вариант» MUST возвращать одну случайную релевантную карточку из города пользователя.
- **FR-013**: «Популярное» MUST показывать нумерованный список из 5 популярных вариантов на ближайшие 30 дней с кнопкой «Показать ещё»; по нажатию «Показать ещё» список MUST дополняться до 10 пунктов (или до исчерпания доступных); каждый пункт MUST открываться в полную карточку.
- **FR-014**: Избранное MUST сохраняться per пользователь Telegram и быть доступно между сессиями.
- **FR-015**: При недоступности ИИ-провайдера подбор MUST работать на правилах и фильтрах по данным афиш; ИИ MAY использоваться для уточнения ранжирования, но не MUST блокировать сценарий.
- **FR-016**: Ответ на любой шаг пользователя MUST укладываться в 5 секунд (p95) при наличии данных в базе.
- **FR-017**: На каждом шаге опроса ответ MUST приниматься только через кнопки быстрого выбора; нажатие кнопки продвигает опрос дальше.
- **FR-018**: Произвольное текстовое сообщение пользователя MUST активировать ИИ-помощника, если оно не является выбором кнопки на текущем шаге опроса; ответ формируется по смыслу запроса (мероприятия, места, рекомендации).
- **FR-019**: Если пользователь отправляет текст во время активного шага опроса, текущий черновик опроса MUST сбрасываться; ИИ-помощник отвечает на запрос без сохранения частично пройденного опроса.
- **FR-020**: Если под фильтры опроса нет подходящих результатов, бот MUST показать сообщение об отсутствии совпадений и кнопки: «Смягчить бюджет», «Сменить формат», «🔄 Заново»; MUST NOT автоматически показывать варианты вне выбранных фильтров без явного действия пользователя.
- **FR-021**: Если событие из источника не содержит категорию активности (`activity_slug`), система MUST при синхронизации: (1) применить rule-based classifier (ключевые слова, тип источника); (2) если категория не определена — вызвать ИИ-помощника с контекстом события (название, описание, источник, город) для выбора одной из категорий опроса: спорт, для детей, семейный отдых, культура, гастро, релакс; (3) сохранить результат в событии перед попаданием в опросный подбор.
- **FR-022**: ИИ-классификация категории MUST выполняться на этапе ingest (синхронизация), а не во время опроса пользователя; опрос MUST оставаться rule-based и укладываться в FR-016.
- **FR-023**: Если ИИ недоступен при sync, система MUST применить только rule-based classifier; события без категории MUST сохраняться с пометкой «не классифицировано» и MUST повторно обрабатываться при следующей синхронизации; sync MUST NOT прерываться из-за недоступности ИИ.

### Key Entities

- **Event / Place**: Мероприятие или место досуга — название, описание, категория активности, дата/период, локация, цена, indoor/outdoor, URL источника, источник(и), признаки аудитории (дети, семья и т.д.).
- **Source**: Внешний сайт-афиша — имя, URL, город покрытия, статус последней синхронизации, тип контента (концерт, спорт, бизнес-форум и т.д.).
- **Survey Session**: Текущий опрос пользователя — выбранная аудитория, категория, бюджет, формат, список уже показанных id.
- **Favorite**: Сохранённая пользователем карточка — ссылка на событие, дата сохранения.
- **User Profile**: Telegram-пользователь — город, история избранного.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь проходит опрос от «Кто идёт?» до карточки результата не более чем за 6 нажатий и 2 минуты wall-clock time.
- **SC-002**: После синхронизации для Москвы в базе доступно не менее 200 уникальных актуальных предложений на горизонте 30 дней из не менее 5 источников.
- **SC-003**: Не менее 80% завершённых опросов возвращают хотя бы одну релевантную карточку (не пустой результат).
- **SC-004**: «Другой вариант» показывает новую карточку в 95% случаев, когда в базе есть ≥2 подходящих варианта.
- **SC-005**: 90% пользователей, прошедших опрос, могут без инструкции понять цену, формат (улица/крыто) и как добраться из карточки.
- **SC-006**: Доля дубликатов одного события в выдаче «Популярное» — 0% (один пользователь не видит одно событие дважды в одном списке).

## Assumptions

- MVP новых источников ориентирован на **Москву**; для остальных 14 городов MVP сохраняет текущие агрегаторы до поэтапного расширения.
- Публичные страницы источников остаются доступны для чтения; при блокировке или captcha источник временно помечается недоступным без падения бота.
- «Популярность» на первом этапе определяется эвристиками (близость даты, наличие в нескольких источниках, категорийный вес), без прямого доступа к закрытой аналитике площадок.
- «У дома» включает онлайн-события и домашние форматы только если они явно представлены в источниках; генерация вымышленных «домашних» активностей без данных не допускается.
- Главный entry point после выбора города — четыре основные кнопки (Случайный / Опрос / Популярное / Избранное) плюс «⚙️ Настройки»; ИИ-помощник доступен через произвольный текст в чат, без отдельной команды.
- Просмотр по категориям из фичи 001 не входит в главный экран; доступен только если явно запрошен через ИИ-помощника или отдельную команду (решение на этапе планирования).
- Навигация «адрес + навигатор» реализуется ссылкой на картографический сервис с названием и адресом места.
- Бюджетные пороги: «до 1000₽» и «до 3000₽» применяются к стоимости на человека, если в источнике указана цена; при отсутствии цены событие не отсекается, но помечается «уточняйте на сайте».
- Категория активности для опроса: (1) из данных источника; (2) rule-based classifier; (3) ИИ при sync если шаги 1–2 не дали результат; (4) при недоступности ИИ — только rule-based, иначе «не классифицировано» до следующей sync (FR-021, FR-023).
