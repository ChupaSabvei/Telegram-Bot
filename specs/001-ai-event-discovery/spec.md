# Feature Specification: ИИ-помощник для подбора мероприятий

**Feature Branch**: `001-ai-event-discovery`

**Created**: 2026-06-12

**Status**: Draft

**Input**: User description: "Создать бота в Telegram со встроенным ИИ-помощником,
который подбирает мероприятия по категориям. Данные о мероприятиях берутся с
интернет-сайтов. Разработка ведётся командой через GitHub."

## Clarifications

### Session 2026-06-12

- Q: В каком городе бот ищет мероприятия на этапе MVP? → A: При первом
  запуске пользователь выбирает город России; в настройках можно сменить город
  поиска в любой момент.
- Q: На какой период вперёд бот показывает мероприятия по умолчанию? → A: 1
  месяц от текущей даты.
- Q: Какие типы мероприятий показывает бот в MVP? → A: Только очные
  (офлайн) мероприятия в выбранном городе.
- Q: Откуда в MVP берутся мероприятия? → A: Крупные агрегаторы афиш; в
  качестве примера — Яндекс Афиша; допускается подключение других агрегаторов
  (KudaGo, Afisha и аналоги).
- Q: Какие мероприятия по стоимости показывает бот? → A: Все мероприятия
  (бесплатные и платные); фильтр по цене в MVP не предусмотрен.
- Q: Какой целевой SLA ответа бота для MVP? → A: 5 секунд (p95), в
  соответствии с конституцией v1.0.0.
- Q: Нужна ли автоматическая ежедневная синхронизация афиш уже в MVP? → A: Да,
  ежедневная авто-синхронизация обязательна в MVP (FR-004).
- Q: Как работает список мероприятий на этапе MVP (US1)? → A: Кнопки открывают
  карточку мероприятия в боте; US3 входит в состав MVP вместе с US1.
- Q: Как бот обрабатывает некорректный ввод в MVP? → A: Пустой текст, сообщения
  >500 символов и не-текст (стикер/фото) — вежливый отказ с подсказкой.
- Q: Какие user stories обязательны в MVP? → A: US1 + US2 + US3 + US4 (полный
  функционал спецификации).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Подбор мероприятий по категории (Priority: P1)

Пользователь открывает бота в Telegram, выбирает интересующую категорию
(например, «Концерты» или «Выставки») и получает список актуальных мероприятий
с кратким описанием, датой, местом и ссылкой на источник.

**Why this priority**: Это минимально жизнеспособный продукт — без подбора по
категориям бот не выполняет свою основную функцию. Пользователь сразу получает
ценность без необходимости формулировать сложный запрос.

**Independent Test**: Можно полностью протестировать, выбрав любую категорию и
проверив, что бот возвращает непустой список предстоящих мероприятий с
обязательными полями (название, дата, место, ссылка).

**Acceptance Scenarios**:

1. **Given** пользователь впервые открыл бота, **When** отправляет команду
   старта, **Then** бот приветствует пользователя и предлагает выбрать город
   России для поиска мероприятий.
2. **Given** пользователь выбрал город при первом запуске, **When** завершает
   выбор, **Then** бот сохраняет город и показывает доступные категории
   мероприятий для выбранного города.
3. **Given** пользователь видит список категорий, **When** выбирает категорию
   «Концерты», **Then** бот показывает до 10 ближайших предстоящих концертов
   в выбранном городе с кнопками для открытия карточки каждого мероприятия.
4. **Given** в выбранной категории нет предстоящих мероприятий, **When**
   пользователь выбирает эту категорию, **Then** бот сообщает, что мероприятий
   не найдено, и предлагает выбрать другую категорию.

---

### User Story 2 - Поиск мероприятий через ИИ-помощника (Priority: P2)

Пользователь описывает своими словами, что хочет посетить (например, «джазовый
концерт в эти выходные» или «бесплатная выставка для детей»), и ИИ-помощник
подбирает наиболее подходящие мероприятия из доступной базы.

**Why this priority**: ИИ-помощник — ключевое отличие продукта от простого
каталога. Повышает удобство и точность подбора, но зависит от наличия базы
мероприятий (User Story 1).

**Independent Test**: Можно протестировать, отправив 5 типовых запросов
разной сложности и проверив, что в 4 из 5 случаев бот возвращает хотя бы одно
релевантное мероприятие в первых трёх результатах.

**Acceptance Scenarios**:

1. **Given** пользователь в диалоге с ботом, **When** пишет «хочу на стендап
   в субботу», **Then** бот возвращает список релевантных комедийных
   мероприятий на ближайшую субботу, отсортированных по релевантности.
2. **Given** пользователь отправил неясный запрос («что-нибудь интересное»),
   **When** ИИ не может определить предпочтения, **Then** бот уточняет запрос
   (предлагает выбрать категорию или указать дату/тип мероприятия).
3. **Given** ИИ-помощник временно недоступен, **When** пользователь отправляет
   текстовый запрос, **Then** бот предлагает воспользоваться подбором по
   категориям и не оставляет пользователя без ответа.

---

### User Story 3 - Просмотр подробностей мероприятия (Priority: P3)

Пользователь выбирает мероприятие из списка и видит полную карточку: название,
описание, дату и время, место проведения, категорию и прямую ссылку на
оригинальную страницу источника.

**Why this priority**: Завершает пользовательский путь от поиска до принятия
решения о посещении. Без деталей список малоинформативен.

**Independent Test**: Можно протестировать, выбрав любое мероприятие из списка
и проверив наличие всех полей карточки и работоспособность ссылки на источник.

**Acceptance Scenarios**:

1. **Given** пользователь видит список мероприятий, **When** нажимает на
   конкретное мероприятие, **Then** бот показывает карточку с полным
   описанием, информацией о стоимости (если доступна) и кнопкой-ссылкой на
   оригинальный сайт.
2. **Given** пользователь просматривает карточку, **When** нажимает «Назад»,
   **Then** бот возвращает к предыдущему списку результатов без потери контекста
   поиска.

---

### User Story 4 - Смена города в настройках (Priority: P2)

Пользователь открывает раздел настроек бота и меняет город поиска мероприятий
на другой город России. Все последующие подборы и поиск учитывают новый город.

**Why this priority**: Город влияет на все результаты; без настройки пользователь
не сможет использовать бота при смене локации. Равнозначно по важности с ИИ-
поиском, так как оба расширяют базовый подбор по категориям.

**Independent Test**: Можно протестировать, сменив город в настройках и проверив,
что результаты по той же категории отличаются и соответствуют новому городу.

**Acceptance Scenarios**:

1. **Given** пользователь ранее выбрал город, **When** открывает настройки и
   выбирает другой город России, **Then** бот подтверждает смену и применяет
   новый город ко всем дальнейшим запросам.
2. **Given** пользователь сменил город, **When** запрашивает мероприятия по
   категории или через ИИ, **Then** результаты относятся только к новому городу.

---

### Edge Cases

- Что происходит, если внешний сайт-источник недоступен при обновлении данных?
  Бот показывает последние закэшированные мероприятия и помечает дату
  актуальности данных.
- Как система обрабатывает дублирующиеся мероприятия с разных сайтов?
  Пользователю показывается одна объединённая карточка; дубли скрываются.
- Что если мероприятие уже прошло? Оно не отображается в результатах поиска
  и подборе по категориям.
- Что если мероприятие состоится позже чем через 1 месяц? Оно не отображается
  в выдаче по умолчанию (горизонт поиска — 30 дней).
- Как бот реагирует на пустой или слишком длинный запрос (>500 символов)?
  Бот отклоняет запрос и просит ввести текст до 500 символов или выбрать
  категорию (без обрезки и обработки).
- Что если пользователь отправляет не текст (стикер, фото)? Бот вежливо
  отклоняет и просит описать запрос текстом или выбрать категорию.
- Что если пользователь закрыл бота до выбора города при первом запуске?
  При следующем обращении бот снова предлагает выбрать город, пока выбор не
  сохранён.
- Что если источник содержит онлайн-мероприятие? Оно не попадает в базу и
  выдачу; при парсинге такие события отфильтровываются.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST предоставлять Telegram-бота, доступного пользователям
  через стандартный механизм поиска и запуска ботов в Telegram.
- **FR-002**: Система MUST отображать фиксированный набор категорий мероприятий
  (минимум: концерты, выставки, театр, спорт, образование, другое).
- **FR-003**: При первом запуске пользователь MUST выбрать город России из
  предложенного списка; выбор сохраняется для последующих сессий.
- **FR-003a**: Пользователь MUST иметь возможность сменить город поиска в
  разделе настроек бота в любой момент.
- **FR-003b**: Пользователь MUST иметь возможность выбрать категорию и получить
  список предстоящих очных мероприятий в этой категории для текущего выбранного
  города.
- **FR-003c**: Система MUST показывать только очные (офлайн) мероприятия;
  онлайн-мероприятия, вебинары и стримы в MVP исключаются из выдачи.
- **FR-004**: Система MUST собирать данные о мероприятиях с крупных
  агрегаторов афиш (минимум Яндекс Афиша как один из источников) и обновлять
  их автоматически не реже одного раза в сутки; ручной запуск дополняет, но
  не заменяет ежедневный scheduler в MVP.
- **FR-004a**: Система MUST поддерживать подключение нескольких агрегаторов
  афиш; добавление нового источника не должно требовать изменения логики бота
  или ИИ-помощника.
- **FR-005**: Каждое мероприятие MUST содержать: название, категорию, дату
  (и время, если доступно), место проведения, краткое описание, URL источника,
  стоимость (бесплатно / цена / «уточняйте на сайте», если данные недоступны).
- **FR-005a**: Система MUST показывать как бесплатные, так и платные мероприятия
  без фильтрации по стоимости в MVP.
- **FR-006**: Система MUST принимать текстовые запросы на естественном языке и
  подбирать релевантные мероприятия с помощью ИИ-помощника в рамках текущего
  выбранного города.
- **FR-007**: При недоступности ИИ-помощника система MUST предлагать подбор по
  категориям как альтернативу (graceful degradation).
- **FR-008**: Система MUST показывать карточку мероприятия с полной информацией
  и ссылкой на оригинальный источник.
- **FR-009**: Система MUST исключать из выдачи прошедшие мероприятия и
  мероприятия, дата которых позже чем через 1 месяц от текущей даты.
- **FR-010**: Система MUST дедуплицировать мероприятия, найденные на разных
  источниках, чтобы пользователь не видел повторы.
- **FR-011**: Система MUST отвечать пользователю в течение 5 секунд (p95) с
  момента отправки запроса при типичной нагрузке (один пользователь, БД с
  предзагруженными событиями, стабильная сеть).
- **FR-012**: Система MUST обрабатывать ошибки внешних источников без падения
  и показывать пользователю понятное сообщение.
- **FR-013**: Система MUST отклонять пустой текстовый ввод, сообщения длиннее
  500 символов и не-текстовые сообщения (стикеры, фото) с вежливой подсказкой
  использовать текст или выбрать категорию.

### Key Entities

- **Мероприятие (Event)**: Предстоящее очное событие; атрибуты: название,
  описание, категория, город, дата/время, место проведения (офлайн), стоимость
  (бесплатно/платно/неизвестно), URL источника, название источника, дата
  последнего обновления.
- **Настройки пользователя (User Settings)**: Предпочтения пользователя;
  атрибуты: выбранный город, дата последнего изменения.
- **Категория (Category)**: Тип мероприятия; атрибуты: название, описание для
  пользователя, порядок отображения в меню.
- **Запрос пользователя (User Query)**: Текстовое сообщение или выбор категории;
  атрибуты: текст запроса, выбранная категория (если есть), временная метка.
- **Источник (Event Source)**: Агрегатор афиш (например, Яндекс Афиша); атрибуты:
  название, базовый URL, тип (агрегатор), статус доступности, дата последней
  успешной синхронизации.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь получает список мероприятий по выбранной категории
  менее чем за 5 секунд (p95) в 95% обращений.
- **SC-002**: Не менее 80% тестовых запросов на естественном языке (из
  подготовленного набора из 20 типовых фраз) возвращают хотя бы одно релевантное
  мероприятие в первых трёх результатах.
- **SC-003**: 90% пользователей в пилотном тестировании (5+ человек) успешно
  находят мероприятие по категории с первой попытки без помощи.
- **SC-004**: База содержит не менее 50 актуальных предстоящих очных
  мероприятий (в горизонте 1 месяца) из минимум 2 агрегаторов афиш (включая
  Яндекс Афишу) на момент запуска MVP.
- **SC-005**: При недоступности ИИ-помощника 100% пользователей получают
  альтернативный путь (подбор по категориям) вместо ошибки.

## Assumptions

- Целевая аудитория — русскоязычные пользователи по всей России; город поиска
  выбирается при первом запуске и меняется через настройки.
- Список городов для MVP — крупные города России (точный перечень определяется
  на этапе планирования; минимум 10 городов-миллионников).
- Горизонт поиска по умолчанию — 1 месяц (30 дней) от текущей даты; расширение
  периода — вне scope MVP.
- Только очные (офлайн) мероприятия в выбранном городе; онлайн-события — вне
  scope MVP.
- Основной тип источников — крупные агрегаторы афиш; Яндекс Афиша — обязательный
  источник MVP; допускаются другие агрегаторы (KudaGo, Afisha и аналоги).
- Для MVP достаточно 2–3 агрегаторов; добавление новых источников не блокирует
  запуск, но расширение за пределы агрегаторов — отдельная фича.
- Авторизация пользователей не требуется: бот работает для любого Telegram-
  пользователя без регистрации.
- Сохранение истории поиска и персональных рекомендаций — вне scope MVP.
- Разработка ведётся командой через GitHub (feature branches, PR) — это
  организационное ограничение, не влияющее на пользовательские сценарии MVP.
- ИИ-помощник работает на русском языке; запросы на других языках в MVP не
  поддерживаются.
- Показываются бесплатные и платные мероприятия без фильтра по цене; фильтр
  стоимости в настройках — вне scope MVP.
- Партнёрские ссылки и комиссии с продажи билетов — вне scope MVP.
- Ежедневная автоматическая синхронизация афиш — обязательна в MVP, не
  откладывается на post-MVP.
- MVP включает все user stories: US1 (категории), US2 (ИИ-поиск), US3 (карточка
  мероприятия), US4 (смена города); поток «список → карточка → назад» обязателен.
- Вся пользовательская коммуникация бота — на русском языке.
