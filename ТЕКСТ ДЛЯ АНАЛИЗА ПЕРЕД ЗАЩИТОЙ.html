Afisha AI Bot — Telegram-бот для подбора мероприятий

Подробное описание проекта для защиты: как работает бот, как он создавался, какие технологии используются, почему выбрана такая архитектура и как обеспечивается качество.

Кратко о проекте

Afisha AI — это Telegram-бот, который помогает пользователю найти офлайн-досуг: мероприятия из афиш, подборки по опросу, случайные варианты, популярные события, избранное и свободный ИИ-поиск по текстовому запросу.

1. Проблема и ценность решения

Обычные афиши требуют, чтобы пользователь сам знал, что искать: концерт, театр, выставку, спорт или детское мероприятие. В реальности запрос часто звучит иначе: «куда сходить с семьёй», «что есть на выходных», «хочу недорого и в помещении», «найди бильярд» или «что популярно в Москве». Бот закрывает эту проблему через короткий диалог и ИИ-помощника.

Для пользователя Меньше ручного поиска: бот сразу выдаёт карточку или подборку с датой, ценой, местом и ссылкой.
Для защиты проекта Есть понятная инженерная задача: сбор данных, нормализация, хранение, поиск, Telegram UX и ИИ.
Для развития Архитектура расширяемая: можно добавлять новые города, источники, сценарии и рекомендации.

2. Как работает бот для пользователя

Основной сценарий

  1. Пользователь открывает Telegram-бота @Afisha_ai_bot и отправляет /start.
  2. Если город ещё не выбран, бот просит выбрать город из поддерживаемого списка.
  3. После выбора города появляется главное меню: «Случайный вариант», «Опрос», «Популярное», «Избранное», «Настройки».
  4. Пользователь выбирает подходящий режим, а бот возвращает карточку события или список.

Опросный подбор

Опрос состоит из четырёх шагов. Он сделан кнопками, чтобы пользователь не вводил свободный текст и не получал непредсказуемый сценарий.

ШагВариантыЗачем нужен
Кто идёт?Один, пара, семья, друзьяФильтрует события по аудитории.
АктивностьСпорт, дети, семейный отдых, культура, гастро, релаксПонимает тип досуга.
БюджетБесплатно, до 1000, до 3000, без лимитаИсключает неподходящие по цене события.
ФорматКрытое, всё равно, у домаУчитывает погоду, локацию и формат.

Другие режимы

  • Случайный вариант — бот быстро показывает одну карточку из города пользователя.
  • Популярное — показывает 5 популярных событий и кнопку «Показать ещё» до 10.
  • Избранное — пользователь сохраняет понравившиеся карточки и возвращается к ним позже.
  • Свободный текст — сообщение вроде «джаз на выходных» или «куда сходить на бильярд» обрабатывается ИИ-помощником.
  • Настройки — смена города без перезапуска проекта или ручного вмешательства.

3. Архитектура приложения

Проект построен как модульный Python-сервис. Основная идея: разделить Telegram-логику, парсинг, ИИ и хранение данных, чтобы каждая часть могла развиваться независимо. 

src/
├── bot/        Telegram handlers, keyboards, formatters, services
├── scrapers/   загрузчики внешних афиш и общий runner синхронизации
├── ai/         LLM-клиент, ранжирование, intent, классификация activity_slug
└── storage/    SQLAlchemy models, schemas, repositories, migrations
МодульЧто делаетПочему это важно
src/bot Обрабатывает команды, кнопки, FSM-опрос, карточки, меню. Telegram-логика не смешивается с базой и парсерами.
src/scrapers Получает события из KudaGo, Яндекс Афиши, Timepad, MTS Live, T-Bank, mos.ru и других источников. Новые источники подключаются как отдельные классы.
src/ai Определяет тип запроса, ранжирует события, подбирает места, классифицирует события. ИИ используется там, где правила слабее, но не блокирует базовый сценарий.
src/storage Хранит события, источники, пользователей, избранное, статусы синхронизации. Данные сохраняются между сессиями и могут работать на SQLite или PostgreSQL.

Запуск и фоновые задачи

Точка входа — python -m src.bot.main. При старте приложение инициализирует базу, регистрирует routers aiogram и запускает polling. Для обновления данных используется APScheduler: каждый день в 03:00 выполняется синхронизация источников.

4. Источники данных и парсинг

Бот не хранит вручную составленную афишу. Он агрегирует данные из внешних публичных источников, нормализует их в единый формат EventDTO, фильтрует по горизонту 30 дней, удаляет дубликаты и сохраняет в БД.

Почему используется Crawlee

В проект добавлены crawlee и playwright. Crawlee используется как Node.js-слой для получения HTML там, где обычный httpx может быть недостаточен: динамические страницы, JavaScript-рендеринг, сложная загрузка карточек или антибот-поведение.

  • scripts/crawlee_fetch.mjs запускает CheerioCrawler или PlaywrightCrawler.
  • Для afisha.timepad.ru и afisha.yandex.ru включается браузерный режим Playwright.
  • src/scrapers/crawlee_client.py вызывает Node-скрипт из Python и возвращает HTML в общий pipeline.
  • Если встречается captcha, клиент возвращает ошибку, а не ломает весь sync.

ScrapingBee и антибот-защита

Для некоторых HTML-источников подключён ScrapingBee: он помогает получать страницы через прокси/stealth-режим и снижает риск captcha. В прошлых проверках ScrapingBee позволил получить большой HTML Яндекс Афиши без 401 и без captcha. Это важный практический элемент, потому что учебный бот работает с реальными сайтами, а не только с тестовыми fixture-файлами.

Главный инженерный принцип парсинга: сбой одного сайта не должен останавливать бота. Каждый scraper изолирован: ошибка фиксируется в статусе источника, остальные источники продолжают синхронизацию.

5. Как используется искусственный интеллект

ИИ в проекте подключён через OpenAI-compatible API. Это значит, что код использует библиотеку openai, но провайдер может быть не только OpenAI. В рабочей конфигурации использовалась модель Groq через API-ключ и base URL https://api.groq.com/openai/v1.

Где именно используется ИИ

  1. Свободный текст пользователя. Бот понимает запросы вроде «джаз на выходных» или «куда сходить на бильярд».
  2. Ранжирование событий. Из базы выбираются кандидаты, а LLM помогает выбрать наиболее релевантные event_id.
  3. Рекомендации мест. Для запросов про бильярд, боулинг, пляжи, караоке и т.д. ИИ формирует список мест и ссылки на карты.
  4. Классификация при синхронизации. Если источник не даёт категорию активности, ИИ определяет activity_slug: спорт, дети, семья, культура, гастро или релакс.

Почему ИИ не используется на каждом шаге опроса

Опросный сценарий сделан rule-based: кнопки → фильтры → запрос к БД → карточка. Это быстрее, дешевле и стабильнее. ИИ применяется на этапе ingest или свободного текстового поиска, а не в каждом callback. Так бот укладывается в цель по скорости ответа до 5 секунд и остаётся работоспособным, даже если LLM-провайдер временно недоступен.

Fallback-логика

Если ИИ недоступен, система использует rule-based classifier и fuzzy fallback. Неклассифицированные события сохраняются и повторно обрабатываются при следующей синхронизации. Это лучше, чем терять данные или останавливать весь процесс.

6. Хранение данных

Для доступа к базе используется SQLAlchemy 2 async. В разработке применяется SQLite, в production предусмотрен PostgreSQL, в том числе Supabase. Такой выбор даёт простой локальный запуск и нормальный путь к продакшену.

7. Как проект создавался

Git и ветки

Разработка ведётся через GitHub и feature branches. В репозитории есть основная ветка master, которая используется как production/stable ветка, и testing, которая нужна для фиксации прогресса и более подробной проверки перед переносом в основную ветку. Текущая крупная feature-ветка — 002-multi-source-survey-flow.

Такой подход нужен, чтобы не ломать стабильную версию бота во время активной разработки: экспериментальные изменения сначала проходят через тестовую/feature-ветку, проверяются, затем могут быть перенесены в production.

Spec Kit development

Основная часть приложения проектировалась через Spec Kit: сначала формировались требования, затем план, контракты, модель данных, quickstart и список задач. Это похоже на промышленный подход: перед кодом фиксируется, что именно должно быть реализовано и как это проверять.

АртефактНазначение
spec.mdФункциональные требования, user stories, edge cases, success criteria.
plan.mdАрхитектура, технический подход, структура модулей, риски.
contracts/Контракты формата событий, scraper-интерфейса, survey flow и matcher.
tasks.mdПошаговый dependency-ordered список задач с TDD и чекпоинтами.
quickstart.mdСценарии ручной проверки перед демонстрацией.

Loop code review

Для ревью использовался кастомный skill loop-code-review. Его идея: независимый reviewer-subagent проверяет активные изменения, возвращает findings и оценку, затем исправления валидируются тестами. Цикл повторяется до оценки не ниже 9.5/10 или до явного сообщения, что actionable comments отсутствуют.

Это полезно для защиты: можно объяснить, что качество проверялось не только вручную, но и через повторяемый review-процесс с независимым контекстом.

Headroom / RTK для экономии токенов

В рабочем процессе предусмотрено использование headroom/RTK-команд для сокращения объёма вывода от терминала и экономии токенов при работе с AI-агентом. Например, вместо полного вывода тестов или git diff можно получать компактную версию, где остаётся самое важное. В текущей локальной PowerShell-среде команда rtk не была доступна в PATH, но сама идея процесса сохранена: меньше шума в выводе, больше внимания на ошибках и решениях.

9. Что говорить преподавателю: аргументы “что, зачем и почему”

Почему Telegram-бот

Telegram снижает порог входа: пользователю не нужно устанавливать отдельное приложение или открывать сайт. Бот работает как диалоговый интерфейс и хорошо подходит для быстрых рекомендаций.

Почему aiogram 3

aiogram 3 асинхронный, хорошо подходит под Telegram polling/webhook, поддерживает routers и FSM. Это удобно для сценариев с кнопками, опросом, callback-обработчиками и несколькими независимыми разделами меню.

Почему несколько источников

Один источник ненадёжен: сайт может отдавать captcha, менять HTML или иметь мало событий. Мульти-источники повышают покрытие и делают подбор релевантнее. Для Москвы задан измеримый критерий: минимум 200 уникальных событий на 30 дней из минимум 5 источников.

Почему SQLite + PostgreSQL/Supabase

SQLite удобен для локальной разработки и защиты на ноутбуке. PostgreSQL/Supabase нужен для production: многопользовательская работа, стабильное хранение, резервное копирование и нормальная эксплуатация на сервере.

Почему Groq/OpenAI-compatible API

OpenAI-compatible интерфейс даёт гибкость: можно использовать Groq, OpenAI или другого провайдера без переписывания всей бизнес-логики. Groq выбран как быстрый LLM-провайдер, подходящий для интерактивного ответа в боте.

Почему Crawlee и ScrapingBee

Реальные сайты часто защищены от простых HTTP-запросов или рендерят данные через JavaScript. Crawlee/Playwright помогают получить HTML динамических страниц, а ScrapingBee помогает обходить часть антибот-ограничений. Это делает проект ближе к реальной инженерной задаче, а не к учебному парсеру одного статичного сайта.

Почему Spec Kit

Spec Kit позволил вести разработку не хаотично, а от требований к реализации: сначала user stories и acceptance criteria, потом план и контракты, затем задачи и тесты. Это снижает риск “написали код, но не то, что нужно пользователю”.

Почему loop-code-review

Кастомный review-loop нужен для качества: независимый агент смотрит diff и ищет баги, регрессии, пробелы в тестах и архитектурные риски. Это имитирует командное code review и помогает находить проблемы до демонстрации.
